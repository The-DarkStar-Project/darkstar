Darkstar Documentation

Darkstar is an open-source vulnerability intelligence platform for external attack-surface management, DAST, network vulnerability scanning and internal scans through separate scanner containers.

Open dashboard Start installation

Responsible and Authorized Use

Darkstar includes security tooling for ASM, DAST, network vulnerability scanning and internal scans. Use this tooling only for legitimate security purposes on systems you own or for which you have explicit prior permission.

Not legal advice

This documentation describes the intended use of Darkstar, but it is not legal advice. Users and organizations remain responsible for complying with laws, contracts, bug bounty rules, cloud provider terms and customer agreements.

User Responsibilities

  • Verify the allowed scope, targets, scan windows and scan intensity in advance.
  • Obtain explicit permission from the owner, client or responsible organization.
  • Use production environments only when that is part of the agreed test scope.
  • Treat results, exports, screenshots, secrets and personal data as sensitive information.
  • Limit access to findings and exports to people who need that information.
  • Stop further tests when a target turns out to be out of scope.

Prohibited Use

Darkstar must not be used for activities outside permission or outside the test scope, including:

  • unauthorized scans, exploitation or access to systems;
  • credential stuffing, password attacks or use of compromised credentials;
  • phishing, social engineering, data theft or collecting personal data without a valid basis;
  • denial-of-service, disruption, destructive tests or aggressive load without approval;
  • persistence, lateral movement or attempts to bypass security monitoring.

Out-Of-Scope Vulnerabilities

If a scan touches an out-of-scope vulnerability or system, stop further testing on that target. Record what was found, do not share sensitive details more widely than needed, and follow a responsible disclosure or Coordinated Vulnerability Disclosure process.

Architecture

Darkstar consists of a web/orchestrator, a MariaDB database, scanner tooling, optional Greenbone/OpenVAS services and optional scanner-only workers. The orchestrator stores tenant data, users, policies, scan queues, logs and results. Scanner workers execute scan jobs from their own network position.

DashboardUsers, scans, filters, exports, settings
Orchestrator APIAuth, queue, scanner nodes, logs, lifecycle
Scanner runtimeBBOT, RustScan, Nuclei, OpenVAS, Asteroid, DAST tools
Tenant databasesFindings, ASM assets, endpoint inventory, schedules

Multi-Tenant Model

Each organization works in a tenant context. Users can be linked to one or more tenants. During login, the user selects the tenant when multiple memberships exist. Roles then determine which actions are visible and allowed.

Installation and Startup

The default stack runs through Docker Compose. Use the darkstar profile for the full environment.

git clone --recurse-submodules git@github.com:The-DarkStar-Project/darkstar.git
cd darkstar
cp .env.example .env
docker compose --profile darkstar up -d --build

Then open the dashboard at:

http://localhost:8080

Individual Scanners

For targeted tasks, choose an individual scanner such as nuclei, openvas, rustscan, nikto, wapiti, zap, dalfox, testssl, retirejs or vulnscan.

Attack Surface Management (ASM)

ASM in Darkstar is intended to make external assets and exposure visible. Darkstar uses BBOT for domain and subdomain reconnaissance, DNS and OSINT sources, and combines that with port information from RustScan. Discovered subdomains are stored as potential targets so you can start focused follow-up scans.

Workflow

  1. Start an Attack Surface scan on a root domain or scope file.
  2. Review the Attack Surface page for hosts, subdomains, IPs, ports and technology hints.
  3. Filter subdomains by parent domain or search term.
  4. Select subdomains and start a follow-up scan from the same page.
  5. Export attack-surface data as CSV, XLSX or HTML for scope and exposure reviews.
Scope discipline

ASM can find many targets that are not all within permission. Validate scope before starting active DAST or aggressive scans.

Dynamic Application Security Testing (DAST)

DAST tests running web applications from outside the codebase. Darkstar combines template-based checks, crawlers, fuzzers and web scanners. Results are normalized into Darkstar findings with host, affected item, severity, confidence, summary, remediation and references.

Template-Based

Nuclei runs standard, network and WordPress templates. Asteroid also uses its own Nuclei DAST templates.

Crawling and Discovery

Katana, Gau, httpx and Feroxbuster collect URLs, forms and hidden paths for further checks.

Active Scanners

OWASP ZAP, Wapiti, Nikto, Dalfox and testssl.sh perform targeted web, XSS and TLS checks.

Which DAST Should You Use?

SituationApproach
Fast baseline on stagingStandard Nuclei or ZAP baseline.
Web app with many routesAsteroid normal/aggressive for crawling, URL normalization and template checks.
XSS focusDalfox or Nuclei XSS templates.
TLS or certificatestestssl.sh through the testssl scanner.
Server misconfigurationsNikto, Wapiti and Nuclei misconfiguration templates.

Scan Internal Networks With Containers

Darkstar can connect scanner-only containers to the central orchestrator. This lets a scanner run in an internal network, at a customer site, in a VPN or in the same Docker network namespace as internal services. The orchestrator remains central; the scan runs from the place where the worker has network access.

What The Scanner Worker Does

  • Authenticates with a one-time generated dscan_... token.
  • Sends heartbeats to the orchestrator.
  • Claims queued scan jobs through /api/scanner-workers/jobs/claim.
  • Executes python3 -m darkstar.main with targets, mode and scanner choice.
  • Streams logs back to the Debug page.
  • Writes results to the central Darkstar database.

Activate A Local Scanner Container

First create a scanner node and attach token in the orchestrator container:

docker compose exec darkstar-web python3 -m darkstar.scanner_attach create \
  --name local-scanner \
  --url http://darkstar-web:8080 \
  --network darkstar_vuln_net \
  --max-parallel-jobs 2

The CLI writes the scanner token and worker settings to a protected env file (0600) and prints a docker run --env-file ... command. Copy tokens only to the worker host or a secret manager. For a local Compose worker, put the same values in .env: 

DARKSTAR_ORCHESTRATOR_URL='http://darkstar-web:8080'
DARKSTAR_SCANNER_TOKEN='dscan_...'
DARKSTAR_SCANNER_NAME='local-scanner'
DARKSTAR_WORKER_MAX_PARALLEL='2'

Then start the scanner worker:

docker compose --profile scanner up -d darkstar-scanner

Remote Or Customer-Network Scanner

For a remote network, use the docker run attach command that Darkstar shows in Settings. The worker needs outbound access to the orchestrator API and to the central MariaDB endpoint, or to a VPN/tunnel that provides that connection.

Choose A Scanner In The UI

  1. Open Scan Center.
  2. Enter targets reachable from the worker's internal network.
  3. Choose the right worker under Scanner appliance, or keep Auto.
  4. Start the scan. Logs appear in Debug; results land in the same tenant database.

MFA and SSO

Set Up MFA

  1. Log in and open Settings.
  2. Open Authentication and click Setup MFA.
  3. Scan the QR code with an authenticator app or copy the secret manually.
  4. Enter the 6-digit code and click Enable MFA.
  5. As a tenant admin, you can then enable Require MFA for this organization.

Darkstar uses TOTP. If MFA is required and a user does not yet have MFA, setup is enforced during login before the session is completed.

Set Up SSO

  1. Create an OIDC application/client with your identity provider.
  2. Use this redirect URI on the public Darkstar URL: /api/auth/sso/callback.
  3. Open Settings -> Authentication -> Setup SSO in Darkstar.
  4. Enter the issuer URL, client ID, client secret and optional allowed email domain.
  5. Enable Enable SSO and save.
  6. Test login through the SSO field on the login screen.
  7. Only then enable Require SSO for this organization.
Public URL

Use SSO_REDIRECT_URI when Darkstar runs behind a reverse proxy and the generated callback URL does not match the public URL.

Email Notifications

Darkstar can report scan results by email. The UI configuration controls who receives mail and at which severity, while SMTP is configured through environment variables.

Configure SMTP

SMTP_HOST='smtp.example.org'
SMTP_PORT='587'
SMTP_FROM='darkstar@example.org'
SMTP_USER='darkstar@example.org'
SMTP_PASSWORD='...'
SMTP_TLS='true'

Configure Notifications In The UI

  1. Open Settings -> Email Notifications.
  2. Enable Enable email notifications.
  3. Add one or more recipients.
  4. Choose the minimum severity: critical, high, medium or low.
  5. Choose whether success, failure and stopped scans should also be reported.
  6. Click Save Notifications and optionally Send Test.

If SMTP_HOST is missing, Darkstar skips email delivery and writes this to the logs.

API Keys, Exports and Reporting

Tenant admins can create API keys for integrations, reporting or CI/CD. Always use the lowest role needed and store keys only in a secret manager or CI/CD secret storage. 

Authorization: Bearer dstar_...

Exports

  • Vulnerabilities: CSV, XLSX and HTML report.
  • Attack Surface: CSV, XLSX and HTML report.
  • Debug logs: available per scan in the Debug page and through the API.

FastAPI also exposes OpenAPI metadata through /openapi.json and API docs through /docs.

Sec/DevOps Pipeline

Security testing should be part of the development and release pipeline. Darkstar fits into that pipeline as the central place for DAST, ASM, network vulnerability scans, evidence and exportable reports.

1Code

Unit tests, linting, SAST and secrets scanning.

2Dependencies

OSV, Trivy, Grype or similar SCA checks.

3Build

Container image scanning and config/IaC checks.

4Staging

Darkstar DAST against a temporary staging URL.

5Gate

Block releases for unaccepted critical/high findings.

Recommended Gates

  • New critical findings block the release.
  • High findings require a ticket, owner and target date.
  • Scan reports are stored as build artifacts.
  • DAST runs against staging by default, not against production without approval.
  • API keys are rotated periodically.

Testing and Test Automation

Darkstar uses a layered testing approach: fast unit tests for pure logic, smoke tests for important web app contracts and Playwright tests for browser behavior, responsive documentation and normal application flows. The GitHub Actions workflow runs unit/smoke and Playwright as separate jobs.

Unit Tests

Validate scan payloads, MFA/TOTP helpers, endpoint matching, scanner workers and scanner output parsing.

Smoke Tests

Check /documentation, static assets, OpenAPI, auth boundaries and API contracts for vulnerabilities, ASM, scans, schedules and notifications.

Playwright

Starts a temporary Uvicorn server and tests documentation plus application flows in Chromium. Scan actions are mocked and do not start real scanners.

Install Dependencies

python3 -m pip install -r requirements-dev.txt

Run Unit and Smoke Tests

python3 -m pytest -m "not playwright"

Run Only Smoke Tests

python3 -m pytest -m smoke

Run Playwright Tests

python3 -m playwright install chromium
RUN_PLAYWRIGHT=1 python3 -m pytest -m playwright

Playwright screenshots are stored in test-results/playwright/. The full test documentation is in docs/testing.md.

Operational Runbook

Daily

  • Check failed or stuck scans.
  • Triage new critical/high findings.
  • Check scanner node heartbeats.

Weekly

  • Review vulnerability and attack-surface exports.
  • Update scan schedules and scope.

Monthly

  • Check MFA/SSO enforcement.
