Darkstar Documentation

Darkstar is een open source vulnerability intelligence platform voor externe attack-surface management, DAST, netwerk vulnerability scanning en interne scans via losse scanner containers.

Verantwoord en toegestaan gebruik

Darkstar bevat security tooling voor ASM, DAST, netwerk vulnerability scanning en interne scans. Gebruik deze tooling uitsluitend voor legitieme beveiligingsdoeleinden op systemen waarvan je eigenaar bent of waarvoor je vooraf expliciete toestemming hebt gekregen.

Geen juridisch advies

Deze documentatie beschrijft bedoeld gebruik van Darkstar, maar vervangt geen juridisch advies. Gebruikers en organisaties blijven zelf verantwoordelijk voor naleving van wetgeving, contracten, bug bounty regels, cloudprovider voorwaarden en klantafspraken.

Verantwoordelijkheden van de gebruiker

  • Controleer vooraf de toegestane scope, targets, scanvensters en scanintensiteit.
  • Zorg voor expliciete toestemming van de eigenaar, opdrachtgever of verantwoordelijke organisatie.
  • Gebruik productieomgevingen alleen wanneer dat binnen de afgesproken testscope valt.
  • Behandel resultaten, exports, screenshots, secrets en persoonsgegevens als gevoelige informatie.
  • Beperk toegang tot findings en exports tot personen die deze informatie nodig hebben.
  • Stop verdere tests wanneer een target buiten scope blijkt te vallen.

Niet toegestaan gebruik

Darkstar mag niet worden gebruikt voor activiteiten buiten toestemming of buiten testscope, waaronder:

  • ongeautoriseerde scans, exploitatie of toegang tot systemen;
  • credential stuffing, wachtwoordaanvallen of gebruik van buitgemaakte credentials;
  • phishing, social engineering, datadiefstal of verzamelen van persoonsgegevens zonder grondslag;
  • denial-of-service, verstoring, destructieve tests of agressieve belasting zonder akkoord;
  • persistentie, laterale beweging of pogingen om beveiligingsmonitoring te omzeilen.

Kwetsbaarheden buiten scope

Als een scan een kwetsbaarheid of systeem buiten scope raakt, stop dan met verdere tests op dat target. Leg vast wat er is gevonden, deel geen gevoelige details breder dan nodig en volg een responsible disclosure of Coordinated Vulnerability Disclosure proces.

Architectuur

Darkstar bestaat uit een web/orchestrator, een MariaDB database, scanner tooling, optionele Greenbone/OpenVAS services en optionele scanner-only workers. De orchestrator bewaart tenantdata, gebruikers, policies, scanqueues, logs en resultaten. Scanner workers voeren scanjobs uit vanaf hun eigen netwerkpositie.

DashboardGebruikers, scans, filters, exports, settings
Orchestrator APIAuth, queue, scanner nodes, logs, lifecycle
Scanner runtimeBBOT, RustScan, Nuclei, OpenVAS, Asteroid, DAST tools
Tenant databasesFindings, ASM assets, endpoint inventory, schedules

Multi-tenant model

Elke organisatie werkt in een tenantcontext. Gebruikers kunnen aan een of meer tenants gekoppeld zijn. Bij login kiest de gebruiker de tenant wanneer er meerdere memberships bestaan. Rollen bepalen daarna welke acties zichtbaar en toegestaan zijn.

Installatie en starten

De standaard stack draait via Docker Compose. Gebruik het `darkstar` profiel voor de volledige omgeving.

git clone --recurse-submodules git@github.com:The-DarkStar-Project/darkstar.git
cd darkstar
cp .env.example .env
docker compose --profile darkstar up -d --build

Open daarna het dashboard op:

http://localhost:8080

Attack Surface Management (ASM)

ASM in Darkstar is bedoeld om externe assets en exposure zichtbaar te maken. Darkstar gebruikt BBOT voor domein- en subdomeinverkenning, DNS en OSINT-bronnen, en combineert dit met portinformatie uit RustScan. Ontdekte subdomeinen worden als potentiele targets bewaard, zodat je gericht vervolgscans kunt starten.

Workflow

  1. Start een Attack Surface scan op een hoofddomein of scopebestand.
  2. Controleer de Attack Surface pagina op hosts, subdomeinen, IP's, poorten en technologie hints.
  3. Filter subdomeinen op parent domain of zoekterm.
  4. Selecteer subdomeinen en start een vervolgscan vanuit dezelfde pagina.
  5. Exporteer attack-surface data als CSV, XLSX of HTML voor scope- en exposure reviews.
Scope discipline

ASM kan veel targets vinden die niet allemaal binnen toestemming vallen. Valideer scope voordat je actieve DAST of agressieve scans start.

Dynamic Application Security Testing (DAST)

DAST test draaiende webapplicaties vanaf buiten de codebase. Darkstar combineert template-based checks, crawlers, fuzzers en webscanners. Resultaten worden genormaliseerd naar Darkstar findings met host, affected item, severity, confidence, summary, oplossing en referenties.

Template-based

Nuclei draait standaard, netwerk en WordPress templates. Asteroid gebruikt daarnaast eigen Nuclei DAST templates.

Crawling en discovery

Katana, Gau, httpx en Feroxbuster verzamelen URLs, formulieren en verborgen paden voor verdere checks.

Actieve scanners

OWASP ZAP, Wapiti, Nikto, Dalfox en testssl.sh voeren gerichte web-, XSS- en TLS-controles uit.

Wanneer gebruik je welke DAST?

SituatieAanpak

Webapp met veel routesAsteroid normal/aggressive voor crawling, URL normalisatie en template checks.
XSS-focusDalfox of Nuclei XSS templates.
TLS of certificatentestssl.sh via de `testssl` scanner.
Server misconfiguratiesNikto, Wapiti en Nuclei misconfiguration templates.

Interne netwerken scannen via containers

Darkstar kan scanner-only containers koppelen aan de centrale orchestrator. Zo kan een scanner draaien in een intern netwerk, op een klantlocatie, in een VPN of in dezelfde Docker network namespace als interne services. De orchestrator blijft centraal; de scan draait vanaf de plek waar de worker netwerktoegang heeft.

Wat de scanner worker doet

  • Authenticeert met een eenmalig aangemaakte `dscan_...` token.
  • Stuurt heartbeats naar de orchestrator.
  • Claimt queued scanjobs via `/api/scanner-workers/jobs/claim`.
  • Voert `python3 -m darkstar.main` uit met targets, mode en scannerkeuze.
  • Streamt logs terug naar de Debug pagina.
  • Schrijft resultaten naar de centrale Darkstar database.

Lokale scanner container activeren

Maak eerst een scanner node en attach token aan in de orchestrator container:

docker compose exec darkstar-web python3 -m darkstar.scanner_attach create \
  --name local-scanner \
  --url http://darkstar-web:8080 \
  --network darkstar_vuln_net \
  --max-parallel-jobs 2

Zet de token en worker instellingen in `.env`:

DARKSTAR_ORCHESTRATOR_URL='http://darkstar-web:8080'
DARKSTAR_SCANNER_TOKEN='dscan_...'
DARKSTAR_SCANNER_NAME='local-scanner'
DARKSTAR_WORKER_MAX_PARALLEL='2'

Start daarna de scanner worker:

docker compose --profile scanner up -d darkstar-scanner

Remote of klantnetwerk scanner

Voor een remote netwerk gebruik je het `docker run` attach command dat Darkstar toont in Settings. De worker heeft outbound toegang nodig naar de orchestrator API en naar de centrale MariaDB endpoint, of naar een VPN/tunnel die die verbinding levert.

Scanner kiezen in de UI

  1. Open `Scan Center`.
  2. Vul targets in die bereikbaar zijn vanuit het interne netwerk van de worker.
  3. Kies bij `Scanner appliance` de juiste worker, of laat `Auto` staan.
  4. Start de scan. Logs verschijnen in `Debug`; resultaten komen in dezelfde tenantdatabase.

MFA en SSO

MFA instellen

  1. Log in en open `Settings`.
  2. Open `Authentication` en klik `Setup MFA`.
  3. Scan de QR-code met een authenticator app of kopieer de secret handmatig.
  4. Voer de 6-cijferige code in en klik `Enable MFA`.
  5. Als tenant admin kun je daarna `Require MFA for this organization` inschakelen.

Darkstar gebruikt TOTP. Als MFA verplicht is en een gebruiker nog geen MFA heeft, wordt bij login eerst setup afgedwongen voordat de sessie wordt afgerond.

SSO instellen

  1. Maak bij je identity provider een OIDC application/client aan.
  2. Gebruik als redirect URI: /api/auth/sso/callback op de publieke Darkstar URL.
  3. Open in Darkstar `Settings` -> `Authentication` -> `Setup SSO`.
  4. Vul de issuer URL, client ID, client secret en eventueel allowed email domain in.
  5. Zet `Enable SSO` aan en sla op.
  6. Test login via het SSO veld op het login scherm.
  7. Schakel pas daarna `Require SSO for this organization` in.
Publieke URL

Gebruik SSO_REDIRECT_URI wanneer Darkstar achter een reverse proxy draait en de automatisch gegenereerde callback URL niet overeenkomt met de publieke URL.

Emailnotificaties

Darkstar kan scanresultaten per email melden. De UI-configuratie bepaalt wie mail krijgt en bij welke severity, terwijl SMTP via environment variabelen wordt ingesteld.

SMTP configureren

SMTP_HOST='smtp.example.org'
SMTP_PORT='587'
SMTP_FROM='darkstar@example.org'
SMTP_USER='darkstar@example.org'
SMTP_PASSWORD='...'
SMTP_TLS='true'

Notificaties in de UI instellen

  1. Open `Settings` -> `Email Notifications`.
  2. Zet `Enable email notifications` aan.
  3. Voeg een of meer ontvangers toe.
  4. Kies de minimale severity: critical, high, medium of low.
  5. Kies of success, failure en stopped scans ook gemeld moeten worden.
  6. Klik `Save Notifications` en daarna eventueel `Send Test`.

Als SMTP_HOST ontbreekt, slaat Darkstar emailverzending over en schrijft dit naar de logs.

API keys, exports en rapportage

Tenant admins kunnen API keys maken voor integraties, rapportage of CI/CD. Gebruik altijd de laagste rol die nodig is en bewaar keys alleen in een secret manager of CI/CD secret storage. 

Authorization: Bearer dstar_...

Exports

  • Vulnerabilities: CSV, XLSX en HTML report.
  • Attack Surface: CSV, XLSX en HTML report.
  • Debug logs: beschikbaar per scan in de Debug pagina en via API.

FastAPI exposeert daarnaast OpenAPI metadata via /openapi.json en API docs via /docs.

Sec/DevOps pipeline

Security testing hoort onderdeel te zijn van de ontwikkel- en releasepipeline. Darkstar past daarin als centrale plek voor DAST, ASM, netwerk vulnerability scans, evidence en exporteerbare rapportages.

1Code

Unit tests, linting, SAST en secrets scanning.

2Dependencies

OSV, Trivy, Grype of vergelijkbare SCA checks.

3Build

Container image scanning en config/IaC controles.

4Staging

Darkstar DAST tegen tijdelijke staging URL.

5Gate

Release blokkeren bij niet-geaccepteerde critical/high findings.

Aanbevolen gates

  • Nieuwe critical findings blokkeren de release.
  • High findings vereisen ticket, eigenaar en streefdatum.
  • Scanrapporten worden als build artifacts bewaard.
  • DAST draait standaard tegen staging, niet tegen productie zonder akkoord.
  • API keys worden periodiek geroteerd.

Operationeel runbook

Dagelijks

  • Mislukte of vastgelopen scans controleren.
  • Nieuwe critical/high findings triageren.
  • Scanner node heartbeats controleren.

Wekelijks

  • Vulnerability en attack-surface exports reviewen.
  • Scanplanning en scope bijwerken.
  • Offline endpoint agents nalopen.

Maandelijks

  • API keys en scanner tokens opschonen.
  • MFA/SSO enforcement controleren.
  • Pipeline artifacts en retention controleren.
